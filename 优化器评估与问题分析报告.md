## 1. 项目概览

### 技术栈

| 层次 | 技术 |
|------|------|
| 框架 | FastAPI 0.116.1 |
| 求解器 | Google OR-Tools 9.9+（CP-SAT 约束规划）|
| 语言 | Python 3.x |
| 部署 | Docker / Uvicorn |
| 入口 | `POST /solve` |

### 两层求解架构

```
POST /solve
  └─ solve_itinerary()
       ├─ Phase 1: solve_joint_assignment()   ← CP-SAT 负责分天 + 选酒店
       └─ Phase 2: solve_day_route()          ← 全排列/2-opt 负责当天点位排序
```

**Phase 1（CP-SAT）**：决定"哪些点放哪天"+ "选哪家酒店"  
**Phase 2（全排列）**：对每天 ≤8 个点尝试所有 n! 排列，分钟级时间轴模拟，找到最优可行排列

## 2. 架构评估

### 优点

- CP-SAT 分天 + 全排列排序的两层解耦设计思路清晰
- `evaluate_order()` 逐分钟模拟时间轴，约束处理细致（营业时间、预约时段、队列、最佳游览窗口）
- 完善的诊断信息输出（交通方式统计、虚拟餐饮插入、fallback 边使用情况）
- 支持多种参数（步速、酒店模式、膳食策略、交通偏好）

### 核心缺陷

| 缺陷 | 严重程度 | 说明 |
|------|----------|------|
| CP-SAT 预算未扣除餐饮时间 | **P0 致命** | 导致系统性超量安排，当天排列全部失败 |
| CP-SAT 预算未估算交通时间 | **P0 致命** | 同上，是超量安排的另一主因 |
| 餐饮约束硬性异常传播 | **P1 严重** | 单日排列失败 → 直接 HTTP 400，无兜底恢复 |
| 午餐窗口过窄（120 分钟）| **P1 严重** | 重型活动后时间轴压缩，窗口根本到不了 |
| 购物中心无餐饮标注 | **P1 严重** | 现实中购物中心均有美食广场，但系统不感知 |
| 晚餐窗口超出 standard 日程 | **P2** | 窗口设 20:30，但 standard 结束于 20:00 |

### Phase 1：CP-SAT 为什么把这 3 个点塞进同一天

CP-SAT 时间预算约束（`app.py:806`）：

```python
day_budget = windows[day].end_minutes - windows[day].start_minutes  # = 660 min
model.Add(
    sum(point_day_load(points[i].point, ...) * x[i][day] for i in range(n))
    <= day_budget * used[day]
)
```

`point_day_load`（`app.py:750`）：

```python
def point_day_load(point: PointIn, day_value: date) -> int:
    return point.suggestedDurationMinutes + PER_POINT_OVERHEAD_MINUTES + queue_minutes_for_day(...)
    #      ──────────────────────────────   ──────────────────────────   ────────────────────────
    #      180 min                          12 min                       0 min（无排队数据）
    #      = 192 min per point
```

**CP-SAT 计算结果**：
```
3 × 192 = 576 min ≤ 660 min (day_budget)
→ CP-SAT 认为可行，分配到同一天 ✓
```

**但实际需要**：
```
活动时长：3 × 192 = 576 min
交通时间：30（酒→景山）+ 43（景山→SKP）+ 20（SKP→国贸）+ 37（国贸→酒）≈ 130 min
          （最优路径下约 93 min 不含返回）
虚拟午餐：45 min（3 个点均非餐厅，必须插虚拟餐）
虚拟晚餐：45 min
─────────────────────
合计：576 + 93 + 45 + 45 = 759 min > 660 min（日程窗口）
```

**CP-SAT 低估了 183 分钟。** 这是根本矛盾所在。

## 5. 设计缺陷总结

### 缺陷 1（P0）：CP-SAT 时间预算公式根本性遗漏

**问题位置**：`app.py:750`（`point_day_load`）和 `app.py:806`（CP-SAT 约束）

```python
# 当前（错误）
day_budget = end_minutes - start_minutes  # e.g. 660 min for standard
point_day_load = suggestedDurationMinutes + 12 + queue_minutes
```

**遗漏项目**：

| 遗漏项 | 典型值（北京城区）| 说明 |
|--------|------------------|------|
| 点间交通时间 | 25-50 分钟/段 | 每增加 1 个点 → 多 2-3 段交通 |
| 虚拟午餐 | 45 分钟 | 当天无餐厅时必须插入 |
| 虚拟晚餐 | 45 分钟 | 同上 |
| 返回酒店 | 15-40 分钟 | 每天结束时的交通 |

**结果**：CP-SAT 把 759 分钟的实际需求当作 576 分钟安排进 660 分钟的日程，造成系统性高估容量。

### 缺陷 2（P0）：全排列失败时直接抛出 HTTP 400，无任何兜底

**问题位置**：`app.py:1551-1563`

```python
except HTTPException as exc:
    if exc.status_code != 400:
        raise
    raise HTTPException(status_code=400, detail={...})  # 直接传播给用户
```

当 Phase 2 排列全部失败时，没有任何机制：
- 回到 Phase 1 要求"减少当天点数"
- 尝试放宽餐饮约束（fallback 到 mealPolicy="off"）
- 自动将超量点位移到下一天

### 缺陷 3（P1）：午餐窗口 12:00–14:00（120 分钟）过窄

`MEAL_SLOT_WINDOWS["lunch"] = (720, 840)`

实际需要插入虚拟午餐的时机：`current_time` 处于 720-795 之间（meal_start + 45 ≤ 840）。  
也就是说 **旅行者必须在 13:15 之前尚未完成所有餐前活动**，才能插入午餐。

而以北京为例，一个 180 分钟的景区从 09:00 出发到完成需要 09:30+192=12:42。加上前往第二个点的交通 43 分钟 = 13:25。虽然 **抵达**时是 13:25（刚超过 840 分钟），但此时已经没有空间插餐了。

### 缺陷 4（P1）：购物中心无法满足餐饮需求

北京SKP 和 国贸商城 都有大型美食广场（餐饮楼层），但 `pointType="shopping"`，不被识别为餐厅。  
系统认为这一天"完全没有餐饮供给"，强制插虚拟餐，而虚拟餐反而因时间紧张插不进去。

**现实情况**：真实游客会在购物中心内用餐，无需专程绕道餐厅。

### 缺陷 5（P2）：晚餐窗口设置与 standard 日程冲突

```python
MEAL_SLOT_WINDOWS["dinner"] = (1080, 1230)  # 18:00–20:30
PACE_MODE_WINDOWS["standard"] = ("09:00", "20:00", 660)  # day_end = 20:00 = 1200
```

`maybe_insert_virtual_meal` 检查 `meal_end > day_end_minutes`，其中 `day_end_minutes = 1200`。  
因此晚餐实际有效窗口为 **18:00–20:00**（而非 20:30），比配置少 30 分钟。

若最后一个活动在 19:15 后结束，晚餐就无法插入（19:15 + 45 = 20:00 = 1200 ≤ 1200 刚好卡线）。

### 缺陷 6（P2）：CP-SAT 亲和力计算促成了不合理的组合

景山公园 - 北京SKP 距离 7.7 km，景山公园 - 国贸商城 距离 6.5 km，都超出了 `AFFINITY_DISTANCE_KM = 3.0 km`。  
但北京SKP 和 国贸商城 仅 2.2 km，在同一个亲和力组。

CP-SAT 为了"亲和力省交通"把 SKP+国贸 绑定，同时为了"填满时间预算"拉入了景山公园（576 < 660）——这在时间预算计算错误的放大下，产生了一个在现实中根本无法完成的组合。

## 6. 如何设计才能得到最接近现实的最优旅游线路

### 6.1 核心修复：修正 CP-SAT 有效日程预算

**修改位置**：`app.py:806` CP-SAT 约束、新增 `effective_day_budget()` 函数

```python
# 建议新增函数
def effective_day_budget(
    req: SolveRequest,
    candidate_point_count: int,
    has_restaurant_on_day: bool,
) -> int:
    """计算 CP-SAT 可用的有效预算（扣除餐饮和交通估算）"""
    base = PACE_MODE_WINDOWS[req.paceMode][2]  # pace 总分钟数

    # 扣除餐饮预算
    if req.mealPolicy == "auto":
        required_meals = len(required_meal_slots(req))  # light=1, standard/compact=2
        if not has_restaurant_on_day:
            base -= required_meals * VIRTUAL_MEAL_MINUTES  # 45 min × N

    # 扣除交通估算：每个点间加 30 min 估算（含返回酒店）
    base -= (candidate_point_count + 1) * ESTIMATED_INTER_POINT_TRAVEL_MINUTES  # 新常量 30 min

    return max(0, base)
```

```python
# 新增常量
ESTIMATED_INTER_POINT_TRAVEL_MINUTES = 30  # 城区平均点间交通时间（分钟）
```

**预算重算效果（standard，3 个非餐厅点）**：

| 项目 | 当前 | 修正后 |
|------|------|--------|
| 基础预算 | 660 | 660 |
| 扣除餐饮（2×45）| ✗ | -90 |
| 扣除交通（4段×30）| ✗ | -120 |
| **有效预算** | **660** | **450** |
| 3 点实际负荷 | 576 | 576 |
| **判断** | 576 ≤ 660 ✓（错误通过）| 576 > 450 ✗（正确拒绝）|

CP-SAT 将被迫把 3 个点分配到 2 天，解决根本矛盾。

### 6.2 购物中心美食广场支持（mealCapable 标记）

**后端实体扩展**（`shopping_places` 表新增字段）：

```sql
ALTER TABLE shopping_places ADD COLUMN "hasFoodCourt" boolean NOT NULL DEFAULT false;
```

**Optimizer 端修改**：

```python
class PointIn(BaseModel):
    # 已有字段...
    hasFoodCourt: bool = False  # 新增

# evaluate_order() 中修改餐饮满足判断
point_can_cover_slot = (
    (point.pointType == "restaurant" and slot in point.mealSlots)
    or (point.hasFoodCourt and slot in ("lunch", "dinner"))
)
```

**效果**：北京SKP 和 国贸商城 标注 `hasFoodCourt=True` 后，系统自动识别可在商场内用餐，无需插入虚拟餐，时间预算还原 90 分钟。

### 6.3 扩宽餐饮时间窗口

当前窗口太窄，不符合实际用餐习惯：

```python
# 当前（过窄）
MEAL_SLOT_WINDOWS = {
    "lunch":  (12 * 60, 14 * 60),        # 12:00–14:00，仅 120 分钟
    "dinner": (18 * 60, 20 * 60 + 30),    # 18:00–20:30
}

# 建议修改（更贴近现实）
MEAL_SLOT_WINDOWS = {
    "lunch":  (11 * 60 + 30, 14 * 60 + 30),  # 11:30–14:30，180 分钟
    "dinner": (17 * 60 + 30, 21 * 60),        # 17:30–21:00
}
```

同时修正晚餐窗口与日程窗口的对齐问题：`day_end_limit` 应动态根据 pace 对齐晚餐 slot_end，而非硬截断。

### 6.4 Phase 2 失败时触发自动恢复机制

当 `solve_day_route()` 全排列失败时，不应直接抛出 400，而是执行兜底策略：

```python
# solve_itinerary() 中的 Phase 2 调用处
try:
    route = solve_day_route(...)
except HTTPException as exc:
    if exc.status_code == 400:
        # 兜底 1：降为 mealPolicy="off" 重试
        req_no_meal = req.model_copy(update={"mealPolicy": "off"})
        try:
            route = solve_day_route(req_no_meal, ...)
        except HTTPException:
            # 兜底 2：将当天最重的点移到下一天
            route = solve_day_route_with_fewer_points(...)
```

**三级兜底**：
1. `mealPolicy="off"` 重试（忽略餐饮约束）
2. 移走当天耗时最长的点到下一天
3. 若仍失败，抛出附带诊断信息的 400（现有行为）

### 6.5 软化餐饮约束为 Cost 惩罚（中期方案）

将"虚拟餐无法插入"从 `HTTPException` 改为惩罚项，加入 `evaluate_order()` 的 `effective_cost`：

```python
MEAL_MISS_PENALTY_MINUTES = 120  # 错过餐窗的惩罚代价（分钟当量）

def maybe_insert_virtual_meal_soft(...) -> tuple[int, bool, float]:
    """返回 (current_time, inserted, penalty)"""
    # ... 如果插不进去，返回 (current_time, False, MEAL_MISS_PENALTY_MINUTES)
    # 而不是 raise HTTPException
```

这样排列不会因为餐饮失败而直接失败，而是记录惩罚成本。最优解可能是"接受一个错过的餐"，而不是"整个日程不可行"。

### 6.6 增加旅程类型标签，防止不合理组合

真实旅游中，同一天应有主题一致性（"购物日"、"文化参观日"）。CP-SAT 的亲和力系统目前只考虑地理距离，可引入"类型亲和力"：

```python
# 在 point_can_be_on_day() 或 CP-SAT 目标函数中
POINT_TYPE_AFFINITY = {
    ("spot", "spot"): +20,          # 同类型加分
    ("shopping", "shopping"): +30,  # 购物点强聚合
    ("spot", "shopping"): -10,      # 文化+购物略降分
}
```

景山公园（spot）+ 两个购物中心（shopping）会受到轻微惩罚，促使 CP-SAT 倾向于将文化景点与其他景点分在同一天。

### 6.7 完整的"接近现实"设计原则总结

| 原则 | 当前状态 | 建议方案 |
|------|----------|----------|
| **时间预算现实性** | 只算活动时长，忽略交通和餐饮 | CP-SAT 预算扣除交通估算 + 餐饮 overhead |
| **餐饮柔性** | 硬约束，插不进去就 400 | 软约束 + 惩罚 + 三级兜底恢复 |
| **商场就餐** | 购物中心不能吃饭 | `hasFoodCourt` 标记，视为餐饮场所 |
| **餐饮时间窗** | 12:00–14:00 过窄 | 11:30–14:30，贴近实际 |
| **类型主题性** | 纯地理亲和力 | 引入类型亲和力，避免景点+商场混搭 |
| **失败恢复** | 排列全败 → 立即 400 | 自动降级：去餐 → 减点 → 最终才 400 |
| **大城市行程** | 每天可以塞满 | 动态调整有效预算，避免超载 |
